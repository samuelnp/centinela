#### Findings
- **Draft field is purely additive.** `Feature.Draft bool` (`json:"draft,omitempty"`) diffs cleanly: every touched reader (`schedulableFeatureSet`/`NonBacklogFeatureSet` in backlog.go, `classifyFeature`/`UnmetDependencies` in readiness.go, `Summary` in roadmap.go, `BuildView`/`buildFeatureView` in view.go, `renderFeature` in mdgen_feature.go) gained a single `if f.Draft { ... }` branch that returns/continues before falling through to the pre-existing, untouched logic. Confirmed via `git diff main...HEAD` on each file. Since `Draft` is `omitempty` and always `false` for pre-existing features, every added branch is unreachable for non-draft roadmaps — serialization and derivation are byte-identical to before. Affected spec: roadmap-json-contract.feature — the spec's own last scenario ("existing non-draft roadmap --json output is unchanged") plus the full `go test ./...` run (3298 passed, 43 packages, exit 0, includes the pre-existing json-contract/readiness test suites) verify this holds. Risk: none observed. Suggestion: none needed.
- **Generalized `promote` preserves the Backlog→phase path verbatim.** `Promote()` branches on `doc.findInBacklog` succeeding; `promoteFromBacklog` is an unedited extraction of the prior single-path logic (still requires `--phase`, still moves the entry, still calls the same `preflightArtifacts`/`writeRawRoadmap`/`appendPromotionArtifacts` sequence — the only refactor is factoring the analysis+quality+markdown writes into shared `appendScoreArtifacts`, called identically by both branches). The new `promoteDraftInPlace` branch is additive and only reached when the slug is not a Backlog entry. Affected spec: deferred-findings-roadmap-capture.feature (promote scenarios). Risk: none — confirmed by the full test suite passing, including `internal/roadmap/promote_regression_test.go` and `tests/acceptance/deferred_findings_promote_rejections_test.go`.
- **Draft exemption confined to a single coverage-set hook, confirmed by full-surface grep.** `grep -rn "\.Draft\b" internal/ cmd/` (excluding tests) shows `f.Draft` is read only in: `backlog.go` (the one coverage-set hook, `schedulableFeatureSet`), `readiness.go` (`classifyFeature`), `roadmap.go` (`Summary`), `view.go` (`BuildView`/`buildFeatureView`), `mdgen_feature.go` (render marker), `promote_inplace.go` (clearing the flag), and `draft.go` (`IsDraftFeature`, consulted independently by `cmd/centinela/start_guard.go`). No other reader touches the field — no leakage into unrelated code paths (e.g. `synthesize.go`'s unrelated `Draft(...)` helper is a different package/concept, PROJECT.md synthesis, not `Feature.Draft`). Risk: none. Suggestion: none.
- **`dependencyTargetSet` is behaviorally identical to the old dependency-validation set when no drafts exist.** `dependencyTargetSet` and `NonBacklogFeatureSet` both now delegate to the same `schedulableFeatureSet(r, includeDrafts)` body that existed pre-feature as `NonBacklogFeatureSet`'s inline logic; `includeDrafts` only changes behavior when `f.Draft == true` is present. For any pre-existing (draft-free) roadmap, `dependencyTargetSet(r) == NonBacklogFeatureSet(r)` exactly, so `ValidateDependencies` is unchanged. Affected spec: roadmap-json-contract.feature / roadmap-senior-pm-analysis.feature (implicitly, via existing dependency-validation tests). Risk: none.
- **Minor, non-blocking: error-message wording drift for a slug that exists nowhere in the roadmap.** `deferred-findings-roadmap-capture.feature`'s "Promote a slug not in the Backlog phase is rejected cleanly" scenario documents the message as stating "the slug is not a Backlog finding" (the old `findInBacklog` wording, `"%q is not a Backlog finding"`). With the generalized `Promote`, a slug absent from the entire roadmap now falls through to `promoteDraftInPlace` → `doc.findFeature`, which returns `"feature %q not found in roadmap"` instead — a different but still non-zero-exit, roadmap.json-unchanged rejection. The corresponding acceptance test (`tests/acceptance/deferred_findings_promote_rejections_test.go:55`, `TestDfrc_PromoteSlugNotInBacklog`) only asserts a non-zero exit code and does not pin the exact substring, so it passes unaffected, and the write-nothing guarantee holds. Affected spec/scenario: deferred-findings-roadmap-capture.feature → "Promote a slug not in the Backlog phase is rejected cleanly". Risk: low — no functional break, but the spec's documented wording no longer matches the actual message for the "exists nowhere" sub-case (it still matches for "exists elsewhere but not draft/Backlog", which the new roadmap-crud-add-remove.feature covers with its own "neither a draft nor a Backlog finding" wording). Suggestion: a follow-up could special-case "slug not found anywhere" in `promoteDraftInPlace` to say "is not a Backlog finding" for parity with the older spec's prose, but this is cosmetic and does not block shipping.

#### Recommendation
- SAFE — the draft extension and generalized promote are additive and gated behind `f.Draft`/location branching with no leakage beyond the documented single coverage-set hook plus the three independent readers and the start guard; the Backlog→phase promote path is unchanged; dependency validation is unchanged for draft-free roadmaps; full test suite passes (3298 tests, 43 packages). One cosmetic error-message wording drift noted above is non-blocking.

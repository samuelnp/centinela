- **Done / in-progress rows omit readiness** — `status` alone carries the signal; `readiness` and `blockedBy` are absent (BuildView only populates them for planned rows).
- **Ready vs blocked planned rows** — a planned feature with all deps done is `ready` (no `blockedBy`); with any unmet dep it is `blocked` and `blockedBy` names exactly the unmet deps.
- **`dependsOn` always serialized** — a no-dependency feature emits `"dependsOn":[]` (non-nil slice), never `null`; declared deps are preserved in order.
- **Non-schedulable phase exclusion** — Backlog/Baseline phases are dropped from `roadmap --json` phases and from `counts`, matching `Summary()`/`DeriveReadiness` scoping.
- **`show --json` verbatim asymmetry** — `roadmap show/list --json` dumps the persisted `Roadmap` (retains Backlog/Baseline, carries no derived `status`/`readiness`), unlike `roadmap --json`.
- **`list` alias** — byte-identical to `show` for both `--json` and text.
- **Empty roadmap** — `{"phases":[],"counts":{"planned":0,"inProgress":0,"done":0}}`; `ready --json` → `[]`; `show --json` exits 0.
- **Zero-feature schedulable phase** — renders `"features":[]`.
- **`ready --json` nil-coalesce** — nothing ready emits `[]`, never `null`.
- **Byte-stability** — `roadmap`, `ready`, `show` `--json` are byte-identical across consecutive runs (ordered-slice iteration, no map ranging); asserted via exact-bytes comparison.
- **Missing `roadmap.json`** — all three `--json` surfaces and the text-mode `roadmap` exit non-zero, print an error to stderr naming the roadmap path, and emit no partial stdout JSON.
- **Malformed JSON** — rejected by `Load` (invalid syntax) with non-zero exit and no partial stdout.
- **Dependency cycle** — rejected by `Load`/`ValidateDependencies` with non-zero exit and no partial stdout.
- **Text-render golden** — acceptance asserts `roadmap show` text equals `roadmap` text and contains no JSON, but does not pin the exact human-facing string against a stored golden; the `ui.RenderRoadmap` output is covered by pre-existing UI tests.
- **Marshal-error branches** — the `json.MarshalIndent` error returns in the three cmd handlers are unreachable for these value types and remain uncovered (idiomatic defensive checks); no way to force a failure without an injected encoder.
- **Workflow-state race** — `FeatureStatus` reads on-disk workflow state per feature; status is derived twice (status + readiness) but against the same immutable on-disk state in a read-only command, so no consistency risk in practice.

#### Findings

**Verdict A — headless short-circuit precedence (headless wins over explicit confirmation/advisor knobs): CLEAN ADDITIVE GATE. No regression.**

The short-circuit is implemented as a single early-return gated on `config.IsHeadless(cfg)`, placed BEFORE the existing per-knob resolver in both hooks:

- `cmd/centinela/hook_context_review_mode.go:12` — `if config.IsHeadless(cfg) { return false }`, sitting above the untouched `effectiveConfirmationMode` resolver (lines 15-22).
- `cmd/centinela/hook_plan_advisor.go:32` — `if config.IsHeadless(cfg) { return nil }`, sitting above the untouched `loadActiveWorkflows()` / `planadvisor.Directive` loop.

`IsHeadless` (`internal/config/headless.go:22`) returns `false` for the zero-value config with no env set (the byte-identical default). Because the gate is a pure early-return that only fires when `IsHeadless == true`, when headless is OFF control falls through to the pre-existing resolver code unchanged. The precedence is therefore additive, not a rewrite of the confirmation/advisor logic.

Independent confirmation (not speculation):

- Back-compat review prompt: `TestShouldRenderReviewPrompt_HeadlessOffStillRenders` (`cmd/centinela/review_mode_headless_test.go:35`) — `CENTINELA_HEADLESS=""` + `every_step` still renders the prompt. Matches `configurable-step-confirmation-mode.feature` "Default mode prompts for every completed step".
- Profile resolver untouched: `TestShouldRenderReviewPrompt_OutcomeSuppresses`, `TestShouldRenderReviewPrompt_ExplicitEveryStepBeatsOutcome`, `TestEffectiveConfirmationMode_Precedence` (`cmd/centinela/review_mode_profile_test.go`) all pass with headless off — proves `enforcement-profiles.feature` "An explicit confirmation mode overrides the profile default" and the strict/guided/outcome → mode mapping are intact.
- Headless-wins behavior: `TestShouldRenderReviewPrompt_HeadlessBeatsEveryStep` and `TestShouldRenderReviewPrompt_HeadlessConfigSuppresses` confirm the new precedence without contradicting the above (the headless scenarios are a strictly new input dimension).
- Plan advisor: `TestRunHookPlanAdvisor_HeadlessSilent` (`cmd/centinela/plan_advisor_headless_test.go`) proves the headless short-circuit emits nothing AND short-circuits before loading workflows; the headless-off advisor path (`hook_plan_advisor_test.go`) still emits directives — matches `add-plan-advisor-mode.feature`.
- Whole suite: `go test ./...` → 1569 passed in 25 packages (0 failures). The pre-existing confirmation/advisor/profile scenarios are all green alongside the new ones.

Conclusion: headless winning over explicit `step_confirmation_mode` / `plan_advisor_mode` is the documented intent (`headless.go` doc comment) and is realized as a gated, off-by-default early return. No existing scenario regresses.

**Verdict B — `internal/verdict` left UNMAPPED in import_graph: ACCEPTABLE. Consistent with `verify`/`ui`/`roadmap`. Does NOT fail the gate.**

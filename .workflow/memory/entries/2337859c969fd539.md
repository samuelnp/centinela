- Affected spec / scenario: specs/roadmap-parallel-readiness.feature — readiness classification (ready/blocked) and diamond-dependency scenarios.
  Risk: none — `internal/roadmap/view.go`'s `readinessIndex()` calls the existing `DeriveReadiness(r)` (internal/roadmap/readiness.go) and indexes it by name; it does not re-implement or fork the classification logic. `roadmap ready --json` calls the existing `ReadySet(r)`. Both new JSON paths are thin consumers of the single existing readiness source of truth, so there is no risk of the JSON contract drifting from the text-mode readiness semantics already locked by this spec.
  Suggestion: none.

- Affected spec / scenario: `Summary()`-based counts (internal/roadmap/roadmap.go) vs. the new `counts` field in `roadmap --json`.
  Risk: none — `BuildView`'s per-phase loop uses the same `isNonSchedulablePhase` predicate (internal/roadmap/baseline.go) that `Summary()` and `DeriveReadiness()` already use to exclude Backlog/Baseline, and the same `FeatureStatus(name)` function to classify done/in-progress/planned. Counts scoping cannot silently diverge from `Summary()` because both read the identical predicate and status function.
  Suggestion: none.

- Affected spec / scenario: none — command-name collision check for `roadmap show` / `roadmap list`.
  Risk: none — `grep AddCommand(roadmap` across cmd/centinela/*.go shows the existing children of `roadmapCmd` are `brownfield, defer, generate, iterate, promote, ready, validate`; no prior `show` or `list` subcommand exists, so the new `roadmapShowCmd` (aliased `list`) introduces no name collision.
  Suggestion: none.

- Affected spec / scenario: none — `roadmap show --json` vs. `roadmap --json` contract shape.
  Risk: low/informational only — the two JSON contracts are intentionally different shapes (derived/filtered view vs. verbatim persisted `Roadmap`, including Backlog/Baseline and no status/readiness fields), which is correct per the spec's explicit "Design lock" comment block, but is worth flagging as a documentation/consumer-education point since a naive Magallanes integrator could conflate the two. This is already called out explicitly in the spec's header comment and in scenario "roadmap show --json emits the persisted Roadmap verbatim, including non-schedulable phases," so no spec ambiguity exists — noting it only as a non-blocking observation.
  Suggestion: none required now; the docs step should make sure this distinction is called out prominently in generated docs (already planned per the spec).

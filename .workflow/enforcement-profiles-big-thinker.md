### Big-Thinker Report: enforcement-profiles
**Date:** 2026-06-11

#### Problem
Centinela enforces exactly one amount of process: prewrite step-gating, a stop-and-ask between every step, and (via the always-on `strict-subagents-v1` orchestration mode) mandatory subagent evidence. That ceremony is right for a small/local model that needs physical rails but taxes a frontier model and under-serves a quick fix — and when process cost exceeds perceived benefit, users switch Centinela off entirely. The feature introduces named presets (`strict`/`guided`/`outcome`) that scale the *process* axis up or down while the *verification* axis (validate-step gates + claim verification) stays welded on, decoupling "how much scaffolding" from "are the claims trusted" (never).

#### Scope
- **In:** `enforcement_profile` global config knob + constants/normalize/validate; per-feature `--profile` override persisted in workflow state; a pure `ProfileDefaults` knob resolver and an `EffectiveProfile(wf,cfg)` resolver; wiring the resolved profile through the three *process* consumers — prewrite step-gating (`EvaluatePrewrite`/`IsAllowedInStep`), confirmation prompt (`shouldRenderReviewPrompt`), and the orchestration-evidence requirement (`NewWithOrder` setting `OrchestrationMode`); surfacing the effective profile in `centinela status`.
- **Out:** any change to `internal/verify` or `internal/gates` (verification is constant by definition); collapsing the 5-step state machine or the per-step artifact existence checks (`ValidateArtifacts` still runs under all profiles); model-capability auto-selection (the dependent `model-capability-profiles` feature); removing existing knobs (`step_confirmation_mode`, `plan_advisor_mode` stay — a profile only sets their *defaults*).

#### Dependencies & Assumptions
- Builds on the existing config normalize/validate pattern (`workflow_mode.go`, `security_gate.go`) and the existing `step_confirmation_mode` machinery — no new infrastructure.
- `EvaluatePrewrite` already receives `cfg` and the workflow slice; `shouldRenderReviewPrompt` already receives `wf` and `cfg`; `validateOrchestration` already early-returns when `OrchestrationMode != strict-subagents-v1`. All three relaxation points are reachable without new plumbing.
- Assumes the resolver is the single source of truth: step-gating, confirmation, and orchestration-evidence all consult `ProfileDefaults`/`EffectiveProfile`, never re-derive knobs independently.
- Assumes "explicitly set" can be recovered for `step_confirmation_mode` (see Risks — the applyDefaults ordering trap).

#### Risks
| Risk | Impact | Likelihood | Mitigation |
|---|---|---|---|
| Back-compat break: an un-configured upgrade silently changes gating/confirmation/evidence | High | Med | Default profile = **guided** AND guided keeps `strict-subagents-v1` set (so today's evidence requirement is preserved); an invariant test asserts an un-configured project's resolved knobs equal today's hardcoded behavior byte-for-byte. |
| applyDefaults normalizes `step_confirmation_mode` to `every_step` BEFORE profile logic runs → cannot distinguish "explicitly every_step" from "defaulted", breaking precedence (explicit knob must beat profile) | High | High | Do NOT consult `cfg.Workflow.StepConfirmationMode` post-normalize for precedence. Capture the **raw decoded** value (empty string = unset) before `applyDefaults` overwrites it, and resolve confirmation lazily: raw-non-empty → explicit (wins); raw-empty → profile default → hardcoded. Implement via a `RawStepConfirmationMode` shadow field populated in `Load` pre-normalize, or move `step_confirmation_mode` normalization out of `applyDefaults` into the lazy resolver. |
| `outcome` relaxation accidentally relaxes per-step artifact checks | Med | Low | `outcome` only bypasses the `IsAllowedInStep` ordering block in `EvaluatePrewrite`; `ValidateArtifacts` is called independently inside `Workflow.Complete` and takes no profile input — it stays untouched. Pin with a test: `complete` under `outcome` with a missing required artifact still fails. |
| Profile vs `step_confirmation_mode` conflict confuses users | Med | Med | One documented precedence rule; `centinela status` renders the *effective* values. |
| Scope creep into model-capability auto-selection | Med | Low | Explicitly out of scope; this feature only reads config/flag. |
| Touched files exceed 100 lines (G1) | Med | Med | Profile constants/normalize, `ProfileDefaults` table, and resolver each in their own small files in `internal/config`/`internal/workflow`. |

#### Rollout
- **Step 1 (no behavior change):** add `EnforcementProfile` to `WorkflowConfig`; `ProfileStrict/Guided/Outcome` constants + `NormalizeEnforcementProfile` (unknown → guided) + `validateEnforcementProfile` (reject unsupported value, naming the field) wired into `validateConfig`; pure `ProfileDefaults(profile) ProfileKnobs`.
- **Step 2:** `EnforcementProfile` field on the `Workflow` struct; `start --profile <p>` normalizes+persists; `EffectiveProfile(wf,cfg)` resolver (per-feature → global → guided default). Capture raw `step_confirmation_mode` pre-normalize to fix the precedence hazard.
- **Step 3:** wire the three consumers — `EvaluatePrewrite` bypasses `IsAllowedInStep` only when effective profile is `outcome` (no-active-workflow block unchanged); `shouldRenderReviewPrompt` resolves effective mode (explicit raw knob → profile default → hardcoded; outcome→auto suppresses); `NewWithOrder` sets `strict-subagents-v1` only for `strict`/`guided`, leaves it empty for `outcome`.
- **Step 4:** surface effective profile in `centinela status`; add the invariant test (gates + `verify.Verify` fire on the validate step under every profile) and the un-configured-equals-today test.

#### Handoff
- **Next role:** feature-specialist
- **Outstanding questions (resolutions):**
  1. **Back-compat default = `guided`**, and **guided keeps `OrchestrationMode="strict-subagents-v1"`** (i.e. guided's "subagent evidence" row is set to *required*, matching today). Today's behavior is step-gating ON + `every_step` + orchestration-evidence REQUIRED. To be byte-identical with the roadmap's "guided is today's default," the only change vs the plan's table is: guided's orchestration-evidence column flips from "no" to **yes** (only `outcome` leaves it empty). `NewWithOrder` becomes profile-aware: `strict`+`guided` → strict mode; `outcome` → empty. Tests that must stay green: all existing tests asserting `OrchestrationMode == strict-subagents-v1` for default-started features (they start under guided, still strict). Tests to update/add: `NewWithOrder` now takes/derives the effective profile, so its callers and any direct `NewWithOrder(feature, order)` unit tests get a profile argument (default guided keeps current assertions); new tests for outcome → empty orchestration mode. No verify/gates test changes.
  2. **applyDefaults hazard:** capture the raw decoded `step_confirmation_mode` (empty = unset) before `applyDefaults` normalizes it; precedence resolver treats raw-non-empty as the explicit winner, raw-empty falls through to profile default then hardcoded. (Same pattern available for `plan_advisor_mode` if profile-defaulted.)
  3. **Invariant confirmed:** `verify.Verify(feature, step, cfg, deps)` and `gates.RunWithFilter(cfg, filter)`/`RunAll(cfg)` take **no profile parameter**; in `complete.go` `runComplete`, `executeValidation()` + `runClaimVerification()` run unconditionally inside the `current == "validate"` branch with no profile guard. Verification is structurally constant as long as no profile branch is introduced around them. Pin: table test over {strict,guided,outcome} asserting completion is blocked on a failing gate/claim in every case.
  4. **outcome/artifact independence confirmed:** `outcome` only bypasses the `IsAllowedInStep` ordering check inside `EvaluatePrewrite`; the per-step artifact existence checks live in `ValidateArtifacts`, called from `Workflow.Complete` with no profile input — unaffected.

Split into two scenarios: "Guided profile requires subagent orchestration evidence" (back-compat) and "Outcome profile does not require subagent evidence."

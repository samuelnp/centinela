Latest Features

Centinela's current user-facing capabilities focus on keeping setup, execution, validation, and documentation aligned across agent sessions.

Claude + OpenCode

Shared prewrite enforcement, postwrite updates, setup prompts, and workflow context across both integrations.

Roadmap-First Bootstrap

centinela init, roadmap generation, senior PM analysis, roadmap quality evaluation, and centinela roadmap validate.

Five-Step Workflow

plan -> code -> tests -> validate -> docs is the standard delivery flow, with roadmap-defined bootstrap exceptions where required.

Prompt-Driven UX

Auto-start for new feature intent, strict orchestration directives, and a compact status line for active workflows.

Managed Migration

centinela migrate, centinela migrate docs, and centinela migrate setup --agent ... keep managed assets current.

Docs Output

centinela docs validate and centinela docs generate --out docs/project-docs/index.html publish the HTML project presentation.

Getting Started

Use this path to learn the Centinela workflow end to end. After bootstrap prerequisites are complete, standard feature delivery follows the five-step path shown here.

1) Bootstrap the project
centinela init
centinela init --agent both
2) Complete setup and roadmap prerequisites
PROJECT.md
ROADMAP.md
.workflow/roadmap-analysis.md + .json
.workflow/roadmap-quality.md + .json
centinela roadmap
centinela roadmap validate
3) Start and move one feature through the workflow
centinela start my-feature
centinela complete my-feature   # plan -> code
centinela complete my-feature   # code -> tests
centinela complete my-feature   # tests -> validate
4) Validate and publish docs
centinela validate
centinela complete my-feature   # validate -> docs
centinela docs validate
centinela docs generate --out docs/project-docs/index.html

Roadmap Narrative

Current roadmap intent, plus detected feature dependency context.

Roadmap excerpt
# Roadmap

> Centinela is a harness-governance layer for AI coding agents. Its guiding
> principle: **treat every agent failure as an engineering problem to fix
> permanently, not a prompt to retry — make correctness enforced, not requested.**
> Each item below names the agent or team failure mode it permanently fixes.
>
> **Capability-spectrum principle.** Centinela must govern *any* model — from a
> small local model running on a laptop to a frontier reasoning model. The
> weaker the model, the more scaffolding it needs (step-gating, artifact
> templates, confirmations); the stronger the model, the more enforcement
> shifts to outcomes (gates + claim verification at delivery). What never
> varies is independent mechanical verification: no model's claims are
> trusted, ever. **Governance adapts; verification is constant.**
>
> Status is derived from workflow state — run `centinela roadmap` for the live view.
> Phases are ordered by leverage and dependency; scheduling follows the
> `dependsOn` graph, not the phase number.

PROJECT.md excerpt
# Centinela — Project Definition

> This file is the single source of truth for everything project-specific.
> CLAUDE.md defines how to build. This file defines what to build.

## Elevator Pitch

Centinela is a harness-governance layer for AI coding agents: a best-practices and engineering workflow enforcement tool that sits on top of Claude Code / OpenCode and makes discipline *enforced* rather than *requested* — forcing the agent to always plan, write code, write tests, validate, and document, in order, without skipping steps.

## Problem Statement

**Who uses it:** Developers using Claude Code to autonomously generate code.

**Pain it solves:** Claude tends to skip planning, skip writing tests, or jump straight to code without validation and documentation. Centinela enforces the plan → code → tests → validate → docs cycle via Claude hooks, blocking out-of-order file writes and requiring explicit step completion.

## Architecture Choice

**Archetype:** n-tier

**Pattern:** N-Tier Layered Architecture

**Why:** Centinela is a CLI tool with a clear, strict dependency stack: commands (cmd/) orchestrate business logic (internal/workflow, internal/gates), which reads configuration (internal/config). Each layer only depends on layers below it. No circular dependencies. Presentation (internal/ui) is a pure rendering concern. This maps naturally to n-tier.

**Reference:** [architecture-overview.md](docs/architecture/architecture-overview.md)

**G2 rule (layer boundaries):** `cmd/` may import `internal/*`. `internal/workflow` and `internal/gates` (domain) may import the leaf utility packages `internal/config`, `internal/gitdiff`, and `internal/orchestration`. `internal/gates` may additionally import `internal/roadmap` (read-only, for the `roadmap_drift` gate's in-memory ROADMAP.md regeneration), mirroring the `internal/ui` read-only allowance. `internal/ui` may import `internal/workflow`, `internal/gates`, `internal/roadmap`, `internal/insights`, `internal/calibration`, `internal/synthesize`, `internal/reconstruct` (read-only, for rendering types). `internal/config` imports nothing internal. `internal/memory` (domain) may import `internal/config` only. `internal/planadvisor` may also import `internal/memory`. `internal/verify` may import `internal/config`, `internal/evidence`, `internal/orchestration`, and `internal/worktree` (read-only); it must not import `cmd/` or `internal/ui`. `internal/doctor` is the **aggregator** layer: a domain orchestrator for `centinela doctor` that may import `internal/config`, `internal/gates`, `internal/roadmap`, `internal/evidence`, `internal/worktree`, `internal/setup`, and `internal/workflow` (all read-only) to diagnose and safely repair project health; it must not import `cmd/` or `internal/ui` and is itself imported only by `cmd/`. This mirrors the `internal/verify` read-only allowance. `internal/insights` also joins the **aggregator** layer: a read-only analytics aggregator for `centinela insights` that may import the `internal/telemetry` leaf (read-only) and stdlib only; it must not import `cmd/` or `internal/ui` and is itself imported only by `cmd/` (its Report type by `internal/ui` for rendering). `internal/calibration` also joins the **aggregator** layer: a read-only per-model calibration analyzer for `centinela calibrate` that may import the `internal/telemetry` + `internal/config` leaves (read-only) and stdlib only; it must not import `cmd/` or `internal/ui` and is itself imported only by `cmd/` (its Report type by `internal/ui` for rendering). `internal/synthesize` also joins the **aggregator** layer: a read-only archetype-inference + PROJECT.md synthesizer for `centinela synthesize` that may import the `internal/analyze` domain package (read-only) and stdlib only; it must not import `cmd/` or `internal/ui` and is itself imported only by `cmd/` (its `Inference` type by `internal/ui` for rendering). Its sole edge `synthesize → analyze` is allowed; `analyze` never imports `synthesize`, so there is no cycle. `internal/reconstruct` also joins the **aggregator** layer: a read-only spec-corpus reconstructor for `centinela reconstruct` that may import the `internal/analyze` domain package (read-only) and stdlib only; it must not import `cmd/` or `internal/ui` and is itself imported only by `cmd/` (its `Reconstruction` type by `internal/ui` for rendering). Its sole edge `reconstruct → analyze` is allowed; `analyze` never imports `reconstruct`, so there is no cycle. `internal/brownmap` also joins the **aggregator** layer: a read-only brownfield roadmap generator for `centinela roadmap brownfield` that may import `internal/analyze` (domain) + `internal/roadmap` (domain) + `internal/reconstruct` (aggregator) read-only and stdlib only; it must not import `cmd/` or `internal/ui` and is itself imported only by `cmd/` (its `Plan` type by `internal/ui` for rendering). Its edges `brownmap → analyze`/`roadmap` are domain edges (allowed); `brownmap → reconstruct` is aggregator→aggregator, so the aggregator layer's `allow` is extended to `["domain","leaf","aggregator"]` to let aggregators compose; `analyze`/`roadmap`/`reconstruct` never import `brownmap`, so there is no cycle. `internal/gitutil` is a leaf (stdlib + `os/exec`): `origin`-remote / `gh` detection plus the delivery-option matrix for `centinela deliver`; it imports nothing internal and is consumed by `cmd/` (and `internal/ui` read-only for its `Option` type), mirroring `internal/golist`/`internal/importgraph`. `internal/delivery` also joins the **aggregator** layer: a read-only delivery-artifact composer for `centinela deliver --via pr` that renders the PR body and a Keep-a-Changelog line out of evidence Centinela already holds. It is pure (no file I/O — the caller in `cmd/` reads sources from disk and passes the bodies in as strings) and may import `internal/verify` (domain) and `internal/evidence` read-only plus stdlib; it must not import `cmd/` or `internal/ui` and is itself imported only by `cmd/` (its `PRBody`/`ChangelogEntry` types by `internal/ui` for rendering, if ever needed). Its sole non-leaf edge `delivery → verify` is aggregator→domain (allowed via the aggregator layer's `allow: ["domain","leaf","aggregator"]`); `verify`/`evidence` never import `delivery`, so there is no cycle. `internal/teamdashboard` also joins the **aggregator** layer: a read-only team-status aggregator for `centinela dashboard` that may import `internal/workflow` (domain), `internal/roadmap` (domain), the `internal/telemetry` leaf, and `internal/insights` (aggregator) read-only plus stdlib; it must not import `cmd/` or `internal/ui` and is itself imported only by `cmd/` (its `Dashboard` type by `internal/ui` for rendering). Its `workflow`/`roadmap` edges are domain (allowed); its `teamdashboard → insights` edge is aggregator→aggregator (allowed via the aggregator layer's `allow: ["domain","leaf","aggregator"]`); `workflow`/`roadmap`/`telemetry`/`insights` never import `teamdashboard`, so there is no cycle.

This rule is mechanically enforced for the mapped layers by the **`import_graph`** gate (`[gates.import_graph]` in `centinela.toml`): it loads the project's import graph via a language provider and fails `centinela validate` on any import that crosses a forbidden layer boundary. Packages not yet assigned to a layer surface a non-failing warning rather than passing silently. The provider is auto-selected by manifest (`go.mod` → go, `package.json` → node, `pyproject.toml`/`requirements.txt` → python) and the gate self-skips with a non-failing warning when no provider matches, so non-Go projects no longer hard-fail; unsupported languages plug in via a custom `script_command`. The provider seam and its backends live in the new leaf package **`internal/importgraph`** (stdlib + `os/exec` + the `internal/golist` leaf only); `internal/gates` (domain) consumes it like any other leaf.

# Roadmap

> Centinela is a harness-governance layer for AI coding agents. Its guiding
> principle: **treat every agent failure as an engineering problem to fix
> permanently, not a prompt to retry — make correctness enforced, not requested.**
> Each item below names the agent or team failure mode it permanently fixes.
>
> **Capability-spectrum principle.** Centinela must govern *any* model — from a
> small local model running on a laptop to a frontier reasoning model. The
> weaker the model, the more scaffolding it needs (step-gating, artifact
> templates, confirmations); the stronger the model, the more enforcement
> shifts to outcomes (gates + claim verification at delivery). What never
> varies is independent mechanical verification: no model's claims are
> trusted, ever. **Governance adapts; verification is constant.**
>
> Status is derived from workflow state — run `centinela roadmap` for the live view.
> Phases are ordered by leverage and dependency; scheduling follows the
> `dependsOn` graph, not the phase number.

## Phase 0: Bootstrap

- **docs-migration-managed-docs** — managed, incrementally upgradable scaffolded docs via `centinela migrate`.

## Phase 1: Harness Capabilities

- **governed-project-memory** — a structured, git-tracked memory ledger that harvests Centinela's own artifacts (edge-case lessons, gate verdicts, decisions) at step boundaries and recalls the relevant slice into later features via the existing context-injection hook. Governed (reviewable, deterministic, no semantic-store dependency), not fuzzy recall.

## Phase 2: Configurable Model Routing
